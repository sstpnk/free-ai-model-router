# Техническое задание: Free AI Model Router

**Репозиторий:** `sstpnk/free-ai-model-router`  
**Назначение:** ежедневный сбор, проверка, оценка и ранжирование бесплатных API-моделей, пригодных для качественной работы с кодом, с последующей генерацией проверяемого примера конфигурации LiteLLM.

## 1. Цель проекта

Разработать автономный процесс, который ежедневно:

1. обнаруживает известных и новых поставщиков моделей и новые модели;
2. собирает сведения об API-доступе, бесплатности, квотах, контексте и возможностях моделей;
3. проверяет фактическую доступность моделей короткими API-запросами, когда для поставщика настроен ключ;
4. получает независимые показатели качества, прежде всего рейтинг кодовых агентов Artificial Analysis;
5. нормализует названия одной модели у разных поставщиков;
6. рассчитывает итоговую оценку пригодности для программирования;
7. оставляет только бесплатные модели с хорошим или отличным качеством;
8. ранжирует их от лучших к худшим;
9. формирует отчеты и `litellm-config-sample.yaml`;
10. проверяет корректность созданного YAML;
11. публикует изменения в GitHub только при содержательном изменении данных.

Процесс не должен автоматически заменять рабочий конфиг LiteLLM или перезапускать рабочий сервис. Сгенерированный файл является предложением для проверки и последующего ручного применения.

## 2. Основные определения

### 2.1. Поставщик

Сервис, предоставляющий модель через API. Одна и та же базовая модель может быть доступна у нескольких поставщиков с разными идентификаторами, квотами, скоростью и условиями.

### 2.2. Модель

Каноническая модель разработчика, например `GLM-4.7-Flash`. Конкретный API-вариант модели у поставщика считается отдельной точкой доступа.

### 2.3. Бесплатность

Модель считается бесплатной для маршрута только при отсутствии платы за входные, выходные и обязательные сопутствующие операции в пределах заявленной квоты.

Поддерживаемые статусы:

- `verified_free`: бесплатность и доступ подтверждены успешным API-запросом текущего запуска;
- `documented_free`: бесплатность подтверждена официальной документацией, но фактический запрос не выполнен;
- `account_specific_free`: бесплатность наблюдается для конкретного аккаунта, но не гарантирована публичным тарифом;
- `temporary_free`: поставщик прямо указывает временный или экспериментальный бесплатный доступ;
- `trial_credit`: используется стартовый денежный баланс; это не постоянная бесплатная модель;
- `unknown`: данных недостаточно;
- `paid`: модель платная;
- `unavailable`: API недоступен или модель удалена.

В автоматически создаваемый маршрут разрешено включать `verified_free`, `documented_free`, `account_specific_free` и `temporary_free`. Статусы `account_specific_free` и `temporary_free` должны быть явно помечены в отчете и конфиге комментариями. `trial_credit`, `unknown`, `paid`, `unavailable` в маршрут не включаются.

### 2.4. Хорошее и отличное качество

Итоговая оценка задается в процентах относительно эталона:

- `100%`: уровень `ChatGPT-5.6 Sol High`;
- `85-100%`: отличное качество;
- `70-84.99%`: хорошее качество;
- ниже `70%`: модель не включается в маршрут.

Поскольку точная конфигурация эталона может отсутствовать в публичных наборах Artificial Analysis, способ нормализации должен быть конфигурируемым и прозрачно описанным в отчете.

## 3. Границы системы

### Входит в объем

- текстовые модели, доступные через программный API;
- модели, пригодные для кодовых агентов, изменения репозитория, работы с терминалом и вызова инструментов;
- бесплатные тарифы, бесплатные модели, бесплатные дневные квоты и бесплатность конкретного аккаунта;
- OpenAI-совместимые и иные API, поддерживаемые LiteLLM;
- генерация данных, отчетов и примера конфигурации LiteLLM;
- ежедневный запуск в GitHub Actions;
- ручной запуск;
- автоматический коммит только сформированных артефактов.

### Не входит в первую версию

- автоматическое применение конфига на Sagitta;
- автоматический перезапуск рабочего LiteLLM;
- обход капчи, платного доступа, защиты от роботов или ограничений сайта;
- создание массовых аккаунтов;
- ротация аккаунтов для обхода квот;
- использование стартового кредита как постоянного бесплатного тарифа;
- оценка локально запускаемых моделей без публичного API;
- публикация секретов или индивидуальных лимитов аккаунта.

### 4.2. Artificial Analysis

Использовать в следующем порядке:

1. официальный Data API;
2. публичные структурированные данные страницы;
3. разбор HTML только как резервный вариант, если это не нарушает условия использования.

Основной источник оценки кодинга:  
`https://artificialanalysis.ai/agents/coding-agents`

Приоритетный показатель: `Artificial Analysis Coding Agent Index`. Индекс объединяет задачи длительной разработки, работу с терминалом и понимание репозитория. Необходимо также сохранять доступные составляющие рейтинга, агентскую оболочку и настройки запуска, поскольку результат относится не только к модели, но и к сочетанию модели с агентом.

Дополнительные источники Artificial Analysis:

- общий каталог моделей;
- Coding Index модели;
- Intelligence Index;
- скорость вывода;
- время до первого токена;
- цена;
- данные по точкам доступа и поставщикам, если они доступны используемому тарифу API.

Обязательная атрибуция Artificial Analysis должна присутствовать в создаваемом отчете.

### 4.3. Hugging Face

Использовать Hugging Face Hub API и Inference Providers для:

- обнаружения новых открытых моделей;
- поиска моделей с тегами `text-generation`, `code`, `agents`, `tool-use`;
- определения популярности, даты обновления, разработчика и лицензии;
- обнаружения поставщиков инференса;
- сопоставления новых моделей с уже известными каноническими моделями.

Наличие модели в Hugging Face не означает, что она бесплатно доступна через удаленный API. Это только сигнал для дальнейшей проверки поставщиков.

### 4.4. Дополнительные источники обнаружения

Система должна поддерживать ежедневный поиск новых моделей и поставщиков через:

- каталог поддерживаемых поставщиков LiteLLM;
- таблицу стоимости и метаданных моделей LiteLLM;
- каталоги OpenRouter и Hugging Face;
- Artificial Analysis;
- GitHub Releases и журналы изменений крупных поставщиков;
- официальные блоги и RSS/Atom-ленты поставщиков;
- GitHub-поиск новых OpenAI-совместимых API и адаптеров LiteLLM;
- каталоги моделей GitHub Models;
- официальные страницы разработчиков моделей;
- настраиваемый список дополнительных URL и лент.

Новые найденные сущности сначала попадают в `discovered`, а не автоматически в рабочий маршрут.

## 5. Приоритет доверия к источникам

Каждому полю присваивается источник и уровень доверия:

1. `runtime_verified`: подтверждено фактическим API-запросом;
2. `official_api`: получено из официального API поставщика;
3. `official_docs`: официальная документация или тариф;
4. `benchmark_api`: официальный API независимого рейтинга;
5. `benchmark_page`: публичная страница независимого рейтинга;
6. `aggregator`: сторонний каталог;
7. `inferred`: вычислено или выведено косвенно;
8. `manual_override`: ручное правило с обязательным обоснованием.

При конфликте значений выбирается более доверенный и более свежий источник. Конфликт не замалчивается, а записывается в отчет.

Каждое значимое поле должно хранить:

```yaml
value: 1000
source_url: "..."
source_type: official_docs
observed_at: "2026-07-18T06:00:00Z"
confidence: high
```

## 7. Сопоставление моделей

Одна модель может иметь разные имена и варианты регистра. Нужен отдельный слой разрешения идентичности.

Порядок сопоставления:

1. стабильный ID официального API;
2. ID Artificial Analysis;
3. ID Hugging Face;
4. разработчик + точное имя + версия;
5. таблица ручных псевдонимов;
6. нормализованное нечеткое совпадение только как предложение.

Нечеткое совпадение не должно автоматически объединять модели. Сомнительные пары записываются в `reports/unresolved-model-matches.md`.

Следует различать:

- базовую модель;
- Flash, Mini, Lite, Turbo и другие варианты;
- варианты reasoning effort;
- версии по датам;
- дообученные варианты;
- разные квантизации;
- разные агентские оболочки в Artificial Analysis.

## 8. Проверка бесплатности и квот

### 8.1. Документальная проверка

Для каждого поставщика собирать:

- бесплатность входных токенов;
- бесплатность выходных токенов;
- цену кеша;
- обязательную подписку;
- необходимость банковской карты;
- разовый кредит;
- срок действия акции;
- RPD, RPM, TPD, TPM;
- ограничения по числу параллельных запросов;
- отдельность или общность квоты между моделями;
- условие пополнения баланса;
- региональные и аккаунтные различия;
- дату вступления условий в силу.

### 8.2. Фактическая проверка

Если в GitHub Secrets задан ключ поставщика:

1. получить список моделей, если API это позволяет;
2. отправить минимальный запрос;
3. проверить обычный ответ;
4. проверить вызов инструмента на отдельном малом тесте, если модель заявляет поддержку;
5. записать HTTP-код, тип ошибки и задержку;
6. не публиковать тело ответа, заголовки авторизации и секреты;
7. не расходовать заметную часть бесплатной квоты.

Минимальный тест должен быть дешевым, детерминированным и не превышать заданный бюджет токенов.

Рекомендуемые статусы результата:

- `success`;
- `authentication_failed`;
- `rate_limited`;
- `quota_exhausted`;
- `model_not_found`;
- `region_blocked`;
- `provider_unavailable`;
- `timeout`;
- `invalid_response`;
- `not_tested`.

Один временный сбой не должен удалять модель из маршрута. Требуется политика устойчивости:

- первый сбой: предупреждение;
- два последовательных ежедневных сбоя: понижение надежности;
- три последовательных сбоя: исключение из нового sample;
- успешная следующая проверка: восстановление статуса.

Исключения для явных ошибок `model_not_found`, `paid_required` и документально подтвержденного удаления модели могут применяться сразу.

## 9. Оценка качества кодинга

### 9.1. Базовый показатель

При наличии данных использовать Artificial Analysis Coding Agent Index как основной показатель. Он должен иметь больший вес, чем общий Intelligence Index.

Важно: строка рейтинга может относиться к сочетанию:

```text
модель + агентская оболочка + настройки рассуждения
```

Поэтому сохраняются `agent_harness`, `reasoning_setting` и дата оценки.

### 9.2. Рекомендуемая формула

Веса должны задаваться в `config/scoring.yaml`. Начальная формула:

```yaml
weights:
  coding_agent_index: 0.50
  coding_model_index: 0.20
  terminal_bench: 0.10
  repository_understanding: 0.08
  tool_calling: 0.05
  context_utility: 0.03
  endpoint_reliability: 0.03
  latency_and_speed: 0.01
```

Если отдельная составляющая уже входит в Coding Agent Index, нельзя бездумно учитывать ее второй раз. Реализация должна поддерживать две схемы:

- `composite_primary`: Coding Agent Index является главным агрегированным показателем, остальные метрики используются как небольшие корректировки;
- `component_weighted`: индекс пересчитывается из отдельных компонентов, когда они доступны.

По умолчанию использовать `composite_primary`.

### 9.3. Нормализация к 100%

В `config/scoring.yaml` задается:

```yaml
reference:
  name: ChatGPT-5.6 Sol High
  score_percent: 100
  artificial_analysis_model_id: null
  manual_reference_value: null
```

Алгоритм:

1. Если эталон присутствует в том же рейтинге и в сопоставимой конфигурации, его значение принимается за 100%.
2. Если эталона нет, используется вручную зафиксированное опорное значение с датой и обоснованием.
3. Если нет ни одного надежного значения, система не должна придумывать процент. Она публикует сырой рейтинг и помечает нормализованный процент как `null`.
4. Ручная оценка разрешена только через `config/manual-overrides.yaml`.
5. В отчете должна быть показана формула и исходные значения.

### 9.4. Недостающие результаты

Модель без прямой строки Coding Agent Index может получить предварительную оценку по:

- Coding Index;
- Terminal-Bench;
- SWE-Bench и аналогичным кодовым тестам;
- общему Intelligence Index;
- результатам близкой версии модели;
- подтвержденной поддержке инструментов.

Такая оценка получает `ranking_confidence: low` или `medium`. Модель с низкой уверенностью может попасть в отчет, но в маршрут включается только при превышении более строгого порога, например 75%, если это задано конфигурацией.

### 9.5. Штрафы

Настраиваемые штрафы:

- нет вызова инструментов;
- нестабильный структурированный вывод;
- контекст меньше минимального;
- временно бесплатный доступ;
- непрозрачная квота;
- два последовательных сбоя;
- низкая уверенность сопоставления;
- модель доступна только через пробный кредит;
- поставщик не поддерживается LiteLLM напрямую и требует непроверенный адаптер.

Штрафы должны быть видны в отчете, а не скрыты внутри итогового числа.

## 10. Правила отбора в маршрут

Модель включается в `litellm-config-sample.yaml`, если одновременно выполняются условия:

1. API-доступ подтвержден или документирован;
2. статус бесплатности разрешен;
3. итоговая оценка не ниже `70%`;
4. модель подходит для кодинга;
5. поддерживается LiteLLM или проверенный OpenAI-совместимый адаптер;
6. модель не удалена и не устарела;
7. надежность выше минимального порога;
8. известен точный идентификатор модели у поставщика;
9. отсутствует критический конфликт источников;
10. секрет поставщика не требуется в сгенерированном файле в открытом виде.

Сортировка:

1. итоговое качество;
2. надежность;
3. величина бесплатной квоты;
4. размер полезного контекста;
5. поддержка инструментов;
6. задержка и скорость.

## 11. Построение цепочки резервов

Цель не просто отсортировать модели, а собрать устойчивый маршрут.

Правила:

- первая модель: максимальное качество при приемлемой надежности;
- следующие модели должны по возможности принадлежать другим поставщикам;
- две модели с одной общей суточной квотой не считаются независимыми резервами;
- одинаковая базовая модель у разных поставщиков может быть резервом доступности, но не обязательно резервом качества;
- временно бесплатные модели не должны быть единственным маршрутом;
- последняя модель должна быть максимально стабильной из бесплатных доступных вариантов;
- число резервов задается параметром, по умолчанию 3;
- не добавлять одну точку доступа дважды под разными псевдонимами.

Пример логического результата:

```text
основная лучшая бесплатная модель
→ лучший независимый бесплатный поставщик
→ стабильный бесплатный резерв
```

## 12. Генерируемый LiteLLM-конфиг

Файл: `output/litellm-config-sample.yaml`.

Требования:

- не содержит реальных ключей;
- использует `os.environ/VARIABLE_NAME`;
- содержит комментарии с датой, оценкой, статусом бесплатности и источником;
- создает одну публичную точку входа, например `coding-auto`;
- использует отдельные внутренние имена развертываний;
- задает последовательные `fallbacks`;
- не использует случайное распределение между моделями;
- задает разумные тайм-ауты, охлаждение и число повторов;
- учитывает минимальный общий размер контекста;
- содержит только синтаксис, поддерживаемый целевой версией LiteLLM;
- проходит загрузку LiteLLM в проверочном режиме.

Пример структуры:

```yaml
model_list:
  - model_name: coding-primary
    litellm_params:
      model: openai/example-free-model
      api_base: https://example.com/v1
      api_key: os.environ/EXAMPLE_API_KEY
      timeout: 180

  - model_name: coding-fallback-1
    litellm_params:
      model: openrouter/example/model:free
      api_key: os.environ/OPENROUTER_API_KEY
      timeout: 180

router_settings:
  fallbacks:
    - coding-auto:
        - coding-primary
        - coding-fallback-1
  num_retries: 0
  cooldown_time: 300
  allowed_fails: 1
  enable_pre_call_checks: true

litellm_settings:
  drop_params: true
```

Конкретная структура должна быть проверена на актуальной версии LiteLLM. Генератор не должен копировать неподтвержденный пример из документации без теста.

## 14. Адаптеры поставщиков

Единый интерфейс:

```python
class ProviderAdapter(Protocol):
    provider_id: str

    async def discover_models(self) -> list[ProviderModel]: ...
    async def fetch_pricing(self) -> list[PricingRecord]: ...
    async def fetch_limits(self) -> list[LimitRecord]: ...
    async def verify_model(self, model: ProviderModel) -> VerificationResult: ...
    def to_litellm(self, model: ProviderModel) -> LiteLLMDeployment: ...
```

Адаптер должен различать:

- официальный каталог моделей;
- тарифы;
- лимиты;
- фактическую проверку;
- преобразование к LiteLLM.

Ошибка одного адаптера не должна останавливать весь запуск.

## 15. Поведение при сбоях источников

Основной принцип: не заменять хорошие вчерашние данные пустыми сегодняшними.

Для каждого источника хранить:

- последний успешный снимок;
- время последнего успеха;
- число последовательных ошибок;
- контрольную сумму;
- тип ошибки;
- срок допустимой давности.

Если источник недоступен:

1. использовать последний успешный снимок;
2. пометить данные как устаревающие;
3. не менять статус на `paid` или `unavailable` только из-за ошибки парсера;
4. не удалять модель из отчета без подтверждения;
5. создать предупреждение в `sources-health.md`;
6. завершить процесс с частичным успехом, если критические данные все же доступны.

Если сломан основной источник рейтинга, новый sample не должен ухудшать порядок моделей на основании пустых значений.

## 17. GitHub Actions

### 17.1. Расписание

Запуск ежедневно, например в 05:17 UTC:

```yaml
on:
  workflow_dispatch:
  schedule:
    - cron: "17 5 * * *"
```

Не использовать ровное начало часа, чтобы снизить вероятность задержки планового запуска.

### 17.2. Этапы задания

1. checkout;
2. установка фиксированной версии Python;
3. установка зависимостей из lock-файла;
4. восстановление кеша;
5. сбор официальных данных;
6. обнаружение кандидатов;
7. загрузка рейтингов;
8. фактические API-проверки;
9. нормализация;
10. расчет оценок;
11. генерация отчетов;
12. генерация LiteLLM sample;
13. проверка схем JSON/YAML;
14. статическая проверка;
15. тестовая загрузка LiteLLM-конфига;
16. тесты;
17. проверка `git diff`;
18. коммит и push только при изменениях;
19. публикация краткого итога в GitHub Actions Summary.

### 17.3. Права

```yaml
permissions:
  contents: write
```

Другие права по умолчанию отключить.

### 17.4. Коммиты

Формат:

```text
data: daily model update 2026-07-18
```

Не создавать пустые коммиты.

При изменении только временных полей, например времени проверки без изменения результата, коммит не нужен. Для этого данные должны быть разделены на содержательные и оперативные.

- повтор с экспоненциальной задержкой только для временных ошибок;
- общий тайм-аут запуска;
- условные HTTP-запросы через ETag и Last-Modified;
- кеширование;
- уважение `robots.txt` и условий использования;
- идентифицируемый `User-Agent` проекта;
- структурированные журналы;
- детерминированная генерация файлов;
- закрепленные зависимости;

- отсутствие телеметрии без явного согласия;

## 23. Безопасность

- никакого выполнения загруженного со страниц кода;
- никакой загрузки весов моделей;
- ограничение размера ответов;
- проверка типов содержимого;
- защита от подмены URL и SSRF через белый список источников;
- запрет редиректов на неизвестные домены либо их явная проверка;
- маскирование заголовков авторизации;
- минимальные разрешения GitHub Actions;
- Dependabot или аналогичная проверка зависимостей;
- CodeQL по возможности;
- проверка сгенерированного YAML до публикации.

## 24. История данных

Хранить компактную историю изменений, а не полный необработанный ответ каждого источника ежедневно.

Обязательно сохранять:

- дату первого обнаружения;
- дату последнего наблюдения;
- историю статуса бесплатности;
- историю лимитов;
- историю рейтинга;
- историю успешности API-проверок;
- причину включения и исключения из маршрута.

Не раздувать репозиторий: большие исходные снимки хранить как сжатые артефакты GitHub Actions либо сохранять только при изменениях.

## 25. Ручные исключения

`config/manual-overrides.yaml` необходим для случаев:

- модель фактически бесплатна для конкретного аккаунта, но это не отражено в публичном прайсе;
- у поставщика нестандартный идентификатор;
- автоматическое сопоставление ошибочно;
- известна общая квота нескольких моделей;
- рейтинг отсутствует, но есть проверенная ручная оценка;
- модель временно запрещена из-за несовместимости.

Каждое исключение содержит:

```yaml
reason: "..."
added_at: "..."
expires_at: "..."
source: "..."
```

Исключение без срока пересмотра должно вызывать предупреждение.

## 26. Критерии приемки первой версии

Первая версия считается готовой, если:

1. ежедневный процесс успешно запускается в GitHub Actions;
2. поддержаны минимум OpenRouter, OpenCode Zen, z.ai, Groq, Gemini, Mistral, GitHub Models, Cloudflare и Hugging Face;
3. реализован расширяемый реестр остальных поставщиков;
4. данные Artificial Analysis загружаются через официальный API или корректный разрешенный резервный источник;
5. новые модели обнаруживаются минимум через Artificial Analysis, Hugging Face, OpenRouter и LiteLLM;
6. для настроенных ключей выполняются безопасные API-проверки;
7. бесплатность и ограничения имеют источники и уровни доверия;
8. рейтинг нормализуется прозрачно и воспроизводимо;
9. модели ниже 70% не попадают в sample;
10. генерируется цепочка независимых поставщиков;
11. `output/litellm-config-sample.yaml` проходит проверку;
12. отчеты показывают исходные данные, формулу, уверенность и причины отбора;
13. сбой одного источника не уничтожает предыдущие корректные данные;
14. секреты не попадают в репозиторий и журналы;
15. изменения автоматически коммитятся только при содержательном diff;
16. рабочий LiteLLM не изменяется автоматически.

## 28. Источники и методологическая база

При реализации учитывать:

- Artificial Analysis Coding Agent Index состоит из нескольких семейств реальных задач разработки и отражает сочетание модели, агентской оболочки и настроек.
